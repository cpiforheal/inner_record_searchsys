# 病案室可视化索引系统行动方案

## 1. 系统定位

本系统不以“完整电子病历系统”为目标，而是建设一个本地化的病案管理工作台：

```text
纸质病历原封不动
系统记录位置、状态、借阅、问题、抽查和整改过程
```

核心目标是解决四类问题：

```text
1. 快速定位：知道某份病历在哪个柜、哪一行、哪一摞、第几本
2. 流转可查：知道病历是否借出、谁借走、何时归还
3. 风险可见：知道哪些病历缺护理单、记录单、签名、报告等材料
4. 整改留痕：知道哪些病历被抽查、发现了什么问题、是否整改完成
```

第一阶段不要追求扫描全文、OCR 全自动识别、诊断质控或医保智能审核。当前最有价值的是把病案室里原本靠人工记忆和纸条维护的信息，变成一个可检索、可追踪、可复核的本地台账。

## 2. 总体架构建议

建议采用本地部署：

```text
React 前端
本地后端服务
SQLite 数据库
院内局域网访问
定时加密备份
```

推荐形态：

```text
病案室主机
  ├── Web 管理界面
  ├── SQLite 数据库
  ├── 备份目录
  └── 操作日志

院内其他电脑
  └── 通过局域网访问
```

不建议第一版上云，也不建议开放公网访问。病历索引包含姓名、住院号、出院日期等敏感信息，应优先保证“够用、可控、可备份”。

## 3. 模块拆分

最终系统建议分成五个核心模块。

### 3.1 书架地图

这是系统主入口，需要可视化还原真实病案室结构：

```text
房间
柜号
行号
摞号
年月标签
本内顺序
```

交互方式：

```text
点击柜子 -> 查看柜内所有行
点击某一行 -> 查看该行所有摞
点击某一摞 -> 查看该摞病历列表
点击某份病历 -> 查看位置、状态、借阅、问题、整改记录
```

书架颜色建议：

```text
绿色：全部正常在架
黄色：有借出或轻中度问题
红色：存在高风险问题
蓝色：正在整改
灰色：待核对或信息不完整
```

### 3.2 病历检索

支持快速搜索：

```text
姓名
姓名拼音首字母
住院号
出院日期
年月
柜号 / 行号 / 摞号
问题标签
借阅状态
整改状态
```

搜索结果必须直接展示：

```text
患者姓名
住院号
出院日期
当前位置
本内顺序
当前状态
是否存在问题
是否被抽查过
```

建议结果示例：

```text
张某某｜ZY2024010231
位置：3号柜 / 第4行 / 第2摞 / 第17本
状态：在架
风险：缺护理记录单，待整改
```

### 3.3 借阅管理

借阅不能只改一个状态，必须形成完整流水。

每次借出记录：

```text
病历住院号
借阅人
借阅科室
借阅用途
借出时间
预计归还时间
经办人
备注
```

归还记录：

```text
实际归还时间
归还经办人
归还状态
是否发现缺页、散页、污损
备注
```

借阅状态建议：

```text
在架
借出
逾期未还
归还待核对
遗失待查
```

关键规则：

```text
一份病历同一时间只能存在一条未归还借阅记录
借出后书架地图同步显示状态
逾期未还进入风险看板
所有借出、归还、修改都写入操作日志
```

### 3.4 问题病历

问题病历是后续风险管理的重点。建议不要只做一个“问题病历”开关，而是做成可配置、多问题、可整改、可复核的台账。

常见问题类型：

```text
缺护理记录单
缺体温单
缺医嘱单
缺病程记录
缺出院记录
缺手术记录
缺知情同意书
缺检查报告
首页信息不完整
签名缺失
时间记录不连续
装订顺序不规范
页码不完整
医保相关材料缺失
其他
```

每个问题建议记录：

```text
问题类型
问题描述
风险等级
发现人
发现时间
责任科室
整改状态
整改期限
整改说明
复核人
复核时间
关闭时间
```

风险等级建议：

```text
高风险：核心文书缺失，影响医保检查、医疗纠纷、手术安全或病历完整性
中风险：内容不完整、签名缺失、时间不连续、报告未归档
低风险：装订、页码、封面、顺序等形式问题
待核对：尚不能确认是否真实缺失
```

整改状态建议：

```text
待处理
整改中
待复核
已关闭
无法整改
```

### 3.5 抽查整改

上级单位抽查、医保检查、院内质控检查应作为单独事件管理。

抽查批次记录：

```text
检查批次名称
检查单位
检查类型
检查日期
涉及病历
发现问题
整改要求
整改截止日期
整改状态
负责科室
经办人
备注
```

检查类型建议：

```text
医保检查
卫健检查
病案质控
院内自查
纠纷复核
其他
```

抽查结果应能反向关联到病历：

```text
一份病历可以出现在多个检查批次中
一个检查批次可以包含多份病历
检查发现的问题应同步进入问题病历台账
整改完成后需要复核并留痕
```

## 5. 手工录入行动方案

2023 年至今约一千多份病历，建议按“轻量建档”完成第一轮录入。

第一轮只录这些字段：

```text
姓名
住院号
出院日期或出院年月
柜号
行号
摞号
第几本
状态
是否待核对
```

不要第一轮录入：

```text
诊断
费用
病程内容
手术细节
检查结果
护理记录全文
```

推荐工作方式：

```text
两人一组
A 负责翻病历和读信息
B 负责系统录入
每录完一摞立即复核该摞数量
每录完一个月份抽查 5%-10%
```

每摞完成后建议贴标签：

```text
柜号-行号-摞号
例如：C03-R04-P02
```

预计工作量：

```text
单人：1-2 个工作日完成基础录入
双人：0.5-1 个工作日完成基础录入
复核：额外 0.5 个工作日
```

## 6. 分阶段落地计划

### 阶段一：定位索引可用

目标：

```text
能通过姓名或住院号找到具体纸质位置
```

范围：

```text
书架地图
病历索引
基础搜索
在架 / 借出 / 待核对状态
SQLite 存储
```

验收标准：

```text
随机搜索 20 份病历，能在 1 分钟内找到纸质原件
位置准确率达到 95% 以上
```

### 阶段二：借阅闭环

目标：

```text
病历借出、归还、逾期都可追踪
```

范围：

```text
借出登记
归还登记
借阅历史
逾期提醒
借阅用途统计
```

验收标准：

```text
任意借出病历都能查到借阅人、科室、用途、时间和经办人
系统中不存在状态为借出但无借阅记录的病历
```

### 阶段三：问题病历台账

目标：

```text
把缺护理单、缺记录单、签名缺失等问题病历挑出来并持续整改
```

范围：

```text
问题类型
风险等级
整改状态
责任科室
复核关闭
问题标签
```

验收标准：

```text
所有已发现问题病历都能查到问题、责任、状态和整改期限
高风险问题可在看板中单独列出
```

### 阶段四：抽查整改管理

目标：

```text
每次上级检查、医保检查、院内质控都能形成整改闭环
```

范围：

```text
抽查批次
抽查病历
发现问题
整改要求
整改期限
复核结果
```

验收标准：

```text
任意一次检查能导出完整整改清单
任意一份被抽查病历能追溯历史检查记录
```

### 阶段五：风险看板和报表

目标：

```text
检查前快速知道风险集中在哪里
```

范围：

```text
高风险病历数量
未整改问题数量
逾期未还病历
按科室统计问题
按问题类型统计
按检查批次统计
```

验收标准：

```text
能在 5 分钟内生成待整改清单、逾期借阅清单和高风险病历清单
```

## 7. 界面重构方向

当前 Demo 的方向是正确的，但后续应从“展示 Demo”升级为“日常工作台”。

建议首页布局：

```text
顶部：全局搜索、今日借阅、待整改、逾期提醒
左侧：书架地图
右侧：当前位置病历列表
底部或抽屉：病历详情、借阅、问题、抽查记录
```

病历详情页建议使用标签页：

```text
基础信息
位置记录
借阅记录
问题记录
抽查整改
操作日志
```

问题病历页面建议：

```text
按风险等级筛选
按整改状态筛选
按责任科室筛选
按问题类型筛选
支持导出整改清单
```

抽查整改页面建议：

```text
检查批次列表
批次详情
涉及病历
发现问题
整改进度
导出检查台账
```

## 8. 权限和安全

建议至少分三类角色：

```text
管理员：用户、字典、备份、系统设置
病案管理员：录入、修改位置、借阅归还、问题整改
查询用户：只能检索和查看，不允许导出或修改
```

关键安全要求：

```text
登录访问
强密码
自动锁屏或超时退出
操作日志不可随意删除
敏感导出需要权限
数据库定时备份
备份文件加密
不开放公网
```

操作日志至少记录：

```text
登录
查询
新增病历
修改位置
借出
归还
新增问题
关闭问题
导出数据
删除或作废记录
```

## 9. 备份和恢复

备份策略：

```text
每日自动备份 SQLite 数据库
每周生成一份长期备份
每月至少做一次恢复测试
备份文件保存到本机和离线移动硬盘
```

备份命名建议：

```text
archive-system-YYYYMMDD-HHmm.db.enc
```

必须注意：

```text
只有备份文件不够，必须确认能恢复
备份文件也包含敏感信息，不能明文到处拷贝
```

## 10. 向后兼容策略

为了应对未来不确定需求，系统设计要保留三类扩展口：

第一，字典扩展：

```text
新增问题类型、检查类型、借阅用途时，不改代码
```

第二，标签扩展：

```text
遇到无法归类的新情况，先用标签标记
```

第三，备注和附件扩展：

```text
允许给借阅、问题、抽查批次增加备注
未来可扩展为上传整改照片或扫描件
```

不要为了未来所有可能性提前做复杂系统。正确方式是：

```text
核心字段稳定
分类字段可配置
异常情况用标签和备注承接
```

## 12. 最小可用版本范围

如果要尽快投入使用，最小可用版本只做以下功能：

```text
书架地图
病历录入
病历搜索
借阅归还
问题标记
SQLite 保存
Excel 导入导出
本地备份
```

暂缓功能：

```text
全文扫描
OCR 自动识别
AI 病历质控
医保规则判断
患者端查询
云同步
复杂审批流
```

## 13. 最终目标

这个系统最终应成为病案室的“电子地图”和“风险台账”：

```text
找得到
借得清
还得明
问题看得见
整改追得上
检查拿得出
```

只要先把这六件事做好，系统就已经能明显减轻人工翻找、借阅混乱和检查前突击整理的压力。
